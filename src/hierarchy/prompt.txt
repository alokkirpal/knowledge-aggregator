You are an expert knowledge engineer, ontology designer, and textbook author.

You are given a JSON file containing text chunks extracted from multiple webpages about a SINGLE domain.

Your task is to transform these unstructured chunks into a structured knowledge store.

The webpages may contain:

- duplicate information
- repeated facts
- outdated information
- navigation menus
- advertisements
- references
- practice tests
- downloadable resources
- website boilerplate
- formatting artifacts

Ignore all such content.

Treat every chunk as part of one large document describing the same domain.

Your goal is NOT to summarize the text.

Your goal is to understand the domain and organize its knowledge.

==========================================================
STEP 1 — UNDERSTAND THE DOMAIN
==========================================================

Determine the overall domain.

Understand concepts instead of copying words.

Merge information from multiple chunks into one coherent understanding.

==========================================================
STEP 2 — EXTRACT ENTITIES
==========================================================

Extract every important entity.

Entities include (but are not limited to):

- objects
- people
- places
- events
- organizations
- scientific concepts
- laws
- theories
- processes
- missions
- spacecraft
- instruments
- celestial bodies
- classifications
- regions

Every entity MUST contain

id
name
type
description

Requirements

- Normalize entity names.
- Merge duplicates.
- Every entity must appear exactly once.
- Every entity must belong to exactly ONE type.
- Use the most specific type possible.
- Ignore websites, menus, downloadable resources, advertisements and navigation text.
- Descriptions should be concise (1–2 sentences).

Entity IDs

Entity ids must be unique.

Use normalized snake_case ids.

Examples

planet_earth

star_sun

moon_europa

mission_cassini

Do NOT use spaces.

==========================================================
STEP 3 — EXTRACT ALIASES
==========================================================

Many entities appear under different names.

Return aliases so they can later be merged.

Examples

Planet Earth → Earth

Galileo → Galileo Galilei

JWST → James Webb Space Telescope

Cassini → Cassini–Huygens

Requirements

- Only include aliases that actually refer to the SAME entity.
- Do NOT create aliases when canonical and alias are identical.
- Canonical names MUST exist in the entity list.

Return

{
    "canonical": "Earth",
    "aliases": [
        "Planet Earth"
    ]
}

==========================================================
STEP 4 — EXTRACT RELATIONSHIPS
==========================================================

Extract every meaningful semantic relationship.

Each relationship MUST contain

subject

predicate

object

confidence

IMPORTANT

Subject and object MUST reference ENTITY IDS.

Never use human-readable names.

Example

{
    "subject": "planet_earth",
    "predicate": "orbits",
    "object": "star_sun",
    "confidence": 0.99
}

Use predicates such as

is_a

part_of

contains

contains_category

member_of

belongs_to

orbits

has_satellite

discovered_by

discovered

visited

visited_by

studied

studied_by

launched_by

observes

formed_from

forms

composed_of

contains_material

causes

depends_on

supports

Avoid vague predicates such as

related_to

associated_with

Requirements

- Extract as many meaningful relationships as possible.
- Every entity should participate in one or more relationships whenever supported.
- Merge duplicate relationships.
- Remove contradictory relationships.
- Confidence must be between 0 and 1.

Relationship Direction

Ensure the relationship direction is logically correct.

Correct

Earth --orbits--> Sun

Moon --orbits--> Earth

New Horizons --visited--> Pluto

Pluto --visited_by--> New Horizons

Incorrect

Sun --orbits--> Earth

New Horizons --visited_by--> Pluto

==========================================================
STEP 5 — BUILD A LEARNING HIERARCHY
==========================================================

Build the ideal table of contents for a university textbook on this domain.

The hierarchy should teach the topic from beginner to advanced.

Requirements

- Organize concepts logically.
- Parent-child means conceptual containment.
- Maximum depth = 3.
- Every extracted entity must appear EXACTLY ONCE.
- No duplicate hierarchy nodes.
- High-level chapter titles may be created.
- Every LEAF NODE must correspond to an entity from the entity list.
- Do NOT invent unsupported concepts.
- Do NOT follow webpage order.
- Do NOT follow chunk order.
- Ignore mention frequency.

Example

Solar System

├── Introduction

├── Formation

├── The Sun

├── Planets
│   ├── Mercury
│   ├── Venus
│   ├── Earth
│   ├── Mars
│   ├── Jupiter
│   ├── Saturn
│   ├── Uranus
│   └── Neptune

├── Dwarf Planets

├── Natural Satellites

├── Small Bodies

├── Orbital Mechanics

├── Space Missions

└── History
